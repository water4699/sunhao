摘 要
　　随着社区零售数字化进程不断推进，社区团购逐渐成为生鲜及高频日用品流通的重要形态。该模式在组织末端配送、降低交易成本方面具有一定优势，但中小型平台在业务协同、权限管理、订单流转和运营支撑方面仍存在系统化不足。围绕这一问题，本文以软件工程实现为主线，设计并实现了一套基于 Spring Boot 的社区团购管理系统，并在系统主体稳定运行的基础上接入销量预测、商品推荐和用户聚类等智能分析功能，作为运营辅助模块对业务管理提供支持。
　　系统整体采用前后端分离的分层架构。后端以 Spring Boot 2.7 为核心，结合 MyBatis-Plus、MySQL、JWT 和 BCrypt 完成用户认证、商品管理、团购活动管理、订单管理、团长审核、日志记录等主要业务功能；前端基于 Vue 3、Element Plus 和 ECharts 实现管理界面与数据可视化；智能分析部分则采用 Python Flask 与 scikit-learn 构建独立服务，通过 HTTP 接口与 Java 主系统协同工作。
　　在当前样本规模和实验条件下，系统取得了初步可用的结果：集成预测模型的 MAE 为 2.31、RMSE 为 3.07；混合协同过滤推荐的 Precision@5 为 0.423、Recall@5 为 0.281；用户聚类在 K=3 时的轮廓系数为 0.584。系统测试表明，核心业务链路能够正常闭环，普通页面响应时间基本控制在目标范围内，并发访问下整体运行较为稳定。上述结果主要用于验证智能分析模块在当前原型系统中的接入可行性与辅助价值。
　　研究表明，以 Java 主业务系统为核心、以智能分析模块为辅助扩展的设计思路，能够较好满足中小型社区团购平台在日常管理和运营支撑方面的实际需求。本文工作可为社区团购系统的设计实现和后续优化提供一定参考。
　　关键词：社区团购；Spring Boot；协同过滤推荐；集成学习；K-Means聚类；微服务架构
　　

Abstract
　　As community retail becomes increasingly digitalized, community group buying has become an important channel for the circulation of fresh products and high-frequency daily necessities. Although this model helps improve last-mile efficiency and reduce transaction costs, many small and medium-sized platforms still lack integrated support in business coordination, permission control, order circulation, and operational assistance. To address this issue, this thesis takes software engineering implementation as the main line, designs and implements a community group buying management system based on Spring Boot, and further integrates sales forecasting, product recommendation, and user clustering as auxiliary intelligent analysis modules for operational support.
　　The system follows a layered architecture consisting of a Java backend, a Vue frontend, and an independent Python analysis service. The backend provides user authentication, product management, activity management, order processing, leader review, and log recording through Spring Boot, MyBatis-Plus, MySQL, JWT, and BCrypt. The frontend is built with Vue 3, Element Plus, and ECharts. The intelligent analysis service is implemented with Flask and scikit-learn and collaborates with the Java system through HTTP interfaces.
　　Under the current dataset scale and evaluation setting, the system achieved preliminary usable results: the ensemble forecasting model obtained an MAE of 2.31 and an RMSE of 3.07; the hybrid collaborative filtering recommender achieved a Precision@5 of 0.423 and a Recall@5 of 0.281; user clustering reached the best silhouette score of 0.584 when K=3. System tests further indicate that the core business workflow can run correctly, page response time is generally within the expected range, and overall performance remains stable under the tested concurrent load. These results are mainly intended to verify the feasibility and auxiliary value of integrating intelligent analysis into the current prototype.
　　Keywords: Community Group Buying; Spring Boot; Collaborative Filtering Recommendation; Ensemble Learning; K-Means Clustering; Microservice Architecture

第1章 绪论
1.1 研究背景与意义
　　近年来，移动互联网、即时配送和社区零售业态的快速发展，为社区团购提供了现实基础。社区团购依托"线上预售、线下自提"的组织方式，在降低流通成本、提升末端履约效率方面表现出较强优势，已成为生鲜及高频日用品零售中的重要模式之一。
　　从行业发展情况看，社区团购在近几年保持较快增长，头部平台的扩张说明该模式具备较强的市场适应性。但与此同时，中小型平台在运营管理层面仍普遍存在信息化基础薄弱、数据利用不足的问题，尤其在备货决策、商品推荐、用户分层和后台协同管理等方面表现较为突出。
　　具体而言，当前社区团购平台在运营支撑方面主要面临以下几类问题：第一，备货决策往往仍依赖人工经验，缺乏对销量变化趋势的有效判断，容易出现热销商品断货与滞销商品积压并存的现象。第二，推荐策略多停留在人工编排或简单热销排序层面，难以根据用户历史行为给出具有针对性的商品展示结果。
　　第三，平台对不同用户群体的消费特征缺乏系统识别，难以围绕高价值用户、沉默用户和价格敏感用户制定差异化运营策略。第四，订单、活动、团长与库存等信息往往分散在多个功能模块中，缺少统一的数据管理与分析支撑，这在一定程度上制约了社区团购平台的精细化运营能力。
　　综合现有平台与开源方案的情况可以看出，头部平台通常具备较强的数据化运营能力，而现有开源系统多侧重业务流程实现，对运营分析与辅助决策支持关注不足。基于此，本文将研究重点放在中小型社区团购平台场景下，尝试构建兼顾业务管理与运营支撑的系统原型。
　　本文的研究意义主要体现在以下三个方面。
　　第一，从应用研究角度看，本文将销量预测、协同过滤推荐和用户聚类等方法引入社区团购场景，并结合具体业务流程进行实现与验证，有助于丰富智能算法在社区零售领域中的应用案例。
　　第二，从系统建设角度看，本文围绕社区团购平台的实际管理需求完成了原型系统设计，有利于为中小型平台的信息化建设和数据化运营提供可参考的实现思路。
　　第三，从工程实现角度看，本文采用Spring Boot与Python服务协同的方式完成业务系统与算法模块的集成，这对类似场景下的轻量化智能分析接入具有一定参考价值。
1.2 国内外研究现状
　　1.2.1 社区团购平台研究现状
　　社区团购的学术研究起步较晚，但近年来增长迅速。周子祺（2022）分析了社区团购的发展现状与零售业变革趋势，指出社区团购通过预售集采和社区自提模式有效降低了流通成本，但同时也面临供应链管理粗放和用户留存困难等挑战[1]。张建军和赵启兰（2018）从新零售视角研究了流通供应链的商业模式转型，认为数据驱动和技术赋能是零售供应链升级的关键路径[11]。王成荣和王春娟（2020）探讨了后疫情时代智慧型社区商业服务体系的建设，强调数字化工具在社区商业运营中的重要性[12]。彭岚（2020）对新零售下的全渠道整合策略进行了系统述评，指出订单履行效率和信息服务能力是影响用户体验的核心因素[17]。
　　国际上，社区团购对应的学术概念为"Social Commerce"或"Community-Based E-Commerce"。Liang和Turban（2011）较早地界定了社交电商的研究框架[18]。Liang等（2011）进一步研究了社交支持和关系质量对社交电商购买行为的驱动机制，发现社交信任是促进购买意愿的关键因素[16]。Lin等（2017）对社交电商的研究主题和发展趋势进行了系统综述[19]。
　　上述研究从零售模式、供应链管理、社区商业服务等角度为社区团购提供了理论基础，但鲜有研究将机器学习算法系统性地应用于社区团购的运营决策支持，这正是本文的研究切入点。
　　1.2.2 推荐算法研究现状
　　推荐系统是电子商务领域的核心技术之一。协同过滤（Collaborative Filtering，CF）由Goldberg等（1992）首次提出[13]，其核心思想是利用群体行为数据进行个性化推荐。根据计算相似度的维度不同，协同过滤分为基于用户的协同过滤（User-CF）和基于物品的协同过滤（Item-CF）两种基本范式。
　　Sarwar等（2001）在WWW会议上发表了Item-CF的经典论文，系统比较了余弦相似度、调整余弦相似度和皮尔逊相关系数三种相似度度量方法，实验表明Item-CF在稀疏数据集上的表现优于User-CF[2]。Koren等（2009）提出了矩阵分解技术，通过将用户-物品评分矩阵分解为低秩矩阵的乘积，有效缓解了协同过滤的数据稀疏性问题[6]。
　　近年来，混合推荐策略（Hybrid Recommendation）成为研究主流。Burke（2002）将混合推荐归纳为加权、切换、特征组合等七种融合策略[20]。在电商场景中，加权融合是最常用的策略，即将多种推荐方法的评分按权重线性组合。本系统采用User-CF与Item-CF的加权混合策略，在社区团购这一小规模数据场景中，兼顾兴趣发现和关联推荐两种能力。
　　1.2.3 销量预测研究现状
　　时间序列预测是机器学习领域的经典问题。传统方法包括ARIMA模型[21]和指数平滑族方法。Holt（1957/2004）提出的双参数指数平滑法通过水平分量和趋势分量的递推更新实现趋势预测[5]，至今仍是短期预测的基线方法之一。
　　在机器学习方法中，Breiman（2001）提出的随机森林通过Bagging策略集成多棵决策树，在回归和分类任务中均表现稳健[3]。Makridakis等（2018）在M4预测竞赛中系统比较了统计方法和机器学习方法的预测表现，指出单一模型难以在所有类型的时间序列上都取得最优效果，组合策略（combination methods）通常能获得更稳定的预测精度[15]。Chen和Guestrin（2016）提出的XGBoost进一步提升了梯度提升树的性能[22]，但在短序列（30天以内）场景中，随机森林的表现与XGBoost接近且调参成本更低。
　　本系统基于上述研究，采用线性回归、随机森林和Holt双指数平滑的三模型集成策略，以R2分数作为启发式依据分配各模型权重，兼顾了线性趋势捕捉、非线性模式识别和序列惯性建模三种能力。
　　1.2.4 用户聚类研究现状
　　用户分群是精准运营的前提。K-Means是应用最广泛的聚类算法之一，由MacQueen（1967）正式提出。Arthur和Vassilvitskii（2007）提出的K-Means++初始化策略通过概率采样选择初始聚类中心，有效改善了标准K-Means对初始值敏感的问题[8]。聚类数K的确定是K-Means的核心难题，Rousseeuw（1987）提出的轮廓系数为K的选择提供了兼顾簇内凝聚度和簇间分离度的客观评估标准[4]。
　　在电商领域，RFM模型（Recency-Frequency-Monetary）是用户价值分析的经典框架[23]。Cheng和Chen（2009）将RFM特征与聚类方法结合，利用粗糙集理论对电商客户进行价值分群，实验验证了RFM特征在客户分类中的有效性[24]。本系统借鉴RFM思想构建用户特征向量，结合轮廓系数自动选K机制，实现了用户群体的自动划分和画像生成。
　　综上，现有研究分别从社区团购业务模式、推荐算法、销量预测和用户分群等角度开展了较为充分的讨论，但将这些方法与社区团购管理系统进行一体化设计并结合具体业务场景加以验证的研究仍相对有限。基于此，本文拟在系统实现的基础上，引入智能分析模块，探索面向中小型社区团购平台的可行技术路径。
1.3 研究内容与目标

第2章 相关技术与理论基础
　　本章主要介绍系统开发过程中所涉及的关键技术与实现基础，并结合社区团购管理场景说明各项技术在系统中的作用，为后续总体设计与模块实现奠定基础。
2.1 后端技术基础：Spring Boot + MyBatis-Plus + MySQL
　　后端部分是本系统实现的核心，需要同时满足业务逻辑承载、接口组织、权限控制和数据访问等方面的要求。因此，在技术选型上，既要考虑开发效率，也要关注框架生态的稳定性与工程实现的可维护性。
　　Spring Boot 2.7.18 被选为后端核心框架[9]。对于本系统而言，后端需要同时承担接口组织、权限控制、事务处理和数据访问等职责，因此技术选型首先强调生态成熟度、开发效率与工程稳定性。Spring Boot 通过自动配置、内嵌容器和完备的 Starter 体系，能够较好满足管理系统类项目对快速开发与可维护性的要求，也便于与安全组件、数据库访问框架和定时任务机制进行集成。
　　持久层方面，本文选择 MyBatis-Plus 作为主要数据访问框架。社区团购系统包含大量分页查询、多条件筛选和列表检索场景，此类需求更强调 SQL 组织的灵活性与查询逻辑的可控性。相较于偏向领域对象映射的全自动 ORM 方案，MyBatis-Plus 在保持开发效率的同时，更便于根据业务需要编写和维护查询逻辑。
　　在具体实现中，系统大量使用条件构造器完成多条件组合查询，以适应商品管理、活动筛选和订单检索等场景下的动态查询需求。相较于手写SQL或较为复杂的规范查询方式，这种实现能够在保证可读性的同时降低开发成本。
　　数据库方面，本系统选择 MySQL 8.0。结合当前课题的数据规模与业务类型，系统数据主要以标准关系表为主，不涉及复杂文档查询或专门的分析型存储需求。MySQL 与 Spring Boot、MyBatis-Plus 的集成较为成熟，部署和运维门槛也相对较低，因此更符合本课题原型系统的实现条件。
2.2 前端技术栈：Vue 3 + Element Plus + ECharts
　　前端技术栈的选型主要考虑组件库生态和管理后台场景的适配性。
　　Vue 3.4（配合 Vite 5 构建工具）被选为前端框架[10]。鉴于本系统以前后台管理页面为主，界面主要由表单、表格、菜单和数据展示组件构成，因此技术选型更关注组件生态、开发效率与页面组织的清晰度。Vue 3 配合 Element Plus 能够较好适应管理后台场景，也便于在较短开发周期内完成页面搭建与交互实现。
　　在前端实现方面，Vue 3的Composition API有助于将页面状态、表单逻辑和接口调用进行集中组织，便于管理后台页面的维护与复用；Pinia用于统一管理登录状态与购物车数据，Vue Router则承担页面跳转与权限拦截功能。
　　数据可视化方面，本系统选择 ECharts 6.0。该库能够较方便地支持折线图、柱状图、饼图和散点图等常见统计图表，适用于社区团购管理后台中销售趋势、订单分布和热销排行等数据展示需求。结合 Vue 3 的响应式机制，可较容易实现图表数据更新与页面联动。
2.3 智能分析扩展：Python Flask + scikit-learn
　　智能分析模块在本文中被定位为对主业务系统的扩展补充，其设计重点不是替代 Java 主系统，而是在保持后端主体稳定的前提下，为运营管理提供额外的数据分析能力。此处主要说明 Python 端的技术选型依据。
　　Flask 被选为算法微服务的 Web 框架。由于该部分主要承担模型调用、结果封装与接口返回等任务，服务职责较为聚焦，因此采用结构轻量、部署简便的 Flask 能够满足当前扩展分析模块的实现需求。
　　scikit-learn是Python中最成熟的传统机器学习库[7]，基于NumPy和SciPy[14]构建。相较于TensorFlow和PyTorch等深度学习框架，scikit-learn更适合本系统所涉及的线性回归、随机森林和K-Means等传统机器学习任务，且无需GPU支持，部署门槛低。本系统使用了LinearRegression、RandomForestRegressor、KMeans、cosine_similarity、StandardScaler和silhouette_score等组件。
2.4 认证与安全：JWT + BCrypt
　　JWT（JSON Web Token）是一种基于JSON的无状态令牌标准（RFC 7519），由Header、Payload和Signature三部分组成。相比基于Session的有状态认证，JWT的无状态特性天然适配前后端分离和微服务架构（详见5.1.1节的对比分析）。本系统使用JJWT 0.9.1库实现JWT认证，采用HMAC-SHA256签名算法，令牌有效期24小时。
　　密码安全采用BCrypt算法（由Spring Security Crypto模块提供）。BCrypt的特点是每次生成随机盐值，且计算开销可调（通过cost factor参数），能有效防御彩虹表攻击和暴力枚举。
2.5 本章小结
　　本章围绕系统所采用的后端、前端、算法服务与安全机制进行了分析，并从开发效率、生态成熟度、功能适配性等方面说明了相应技术的选型依据。上述技术组合能够满足社区团购管理系统在业务实现、界面交互和智能分析接入方面的基本需求。
　　

第3章 系统需求分析
　　系统需求分析是软件开发的基础环节，直接决定了系统的功能范围和质量目标。本章结合社区团购的一般业务流程、常见平台功能形态以及本课题原型系统的实现边界，从功能需求和非功能需求两个维度对系统进行分析。
3.1 系统功能需求分析
　　结合社区团购场景中的典型业务流程，本系统需要支持三种用户角色：普通用户（浏览商品、下单购买）、团长（组织团购活动、管理社区订单）和管理员（全局管理、数据分析）。系统的功能需求按模块划分如下。
　　3.1.1 用户管理模块
　　用户管理模块是系统的基础模块，负责用户身份的注册、认证和信息维护。具体功能需求包括：
　　（1）用户注册：支持用户名、密码和手机号注册，注册时需验证用户名和手机号的唯一性。密码需经过加密存储，不得以明文形式保存。
　　（2）用户登录：验证用户名和密码，登录成功后签发令牌，令牌有效期为24小时。
　　（3）身份认证：系统需在每个需要登录的请求中验证用户身份，未登录用户应被拦截并提示登录。
　　（4）权限控制：系统需区分普通用户、团长和管理员三种角色，不同角色具有不同的功能权限。管理后台功能仅管理员可访问。
　　（5）个人信息管理：用户可查看和修改个人资料，包括真实姓名、手机号等。
　　3.1.2 商品管理模块
　　商品管理模块负责商品信息的维护和展示。具体功能需求包括：
　　（1）商品CRUD：管理员可创建、编辑和删除商品，设置商品名称、描述、原价、团购价、库存、单位等属性。
　　（2）商品图片：支持商品主图和多张详情图片的上传，图片格式支持JPG、PNG、GIF等。
　　（3）商品上下架：管理员可控制商品的上下架状态，下架商品不在前台展示。
　　（4）商品查询：支持按名称关键字搜索、按分类筛选和分页浏览。
　　（5）热销排行：系统自动统计商品销量，支持按销量排序展示热销商品。
　　（6）库存预警：当商品库存低于阈值时，系统能够识别并展示低库存商品列表。
　　3.1.3 分类管理模块
　　分类管理模块负责商品分类的组织和管理，支持分类的增删改、两级树形结构（父分类-子分类）以及启用/禁用状态控制。
　　3.1.4 团购活动模块
　　团购活动模块是社区团购系统的核心业务模块。具体功能需求包括：
　　（1）活动创建：团长或管理员可创建团购活动，设置活动名称、描述、起止时间和最低成团人数。
　　（2）活动商品配置：为活动添加商品并设定活动价格和活动库存，活动价通常低于商品的日常团购价。
　　（3）活动状态管理：活动需支持未开始、进行中、已结束三种状态的流转。只有进行中的活动接受用户下单。
　　（4）活动查询：用户可浏览当前进行中的活动列表和活动详情。
　　3.1.5 订单管理模块
　　订单管理模块负责订单的全生命周期管理。具体功能需求包括：
　　（1）订单创建：用户选择活动商品后提交订单，系统需验证活动状态、进行库存预检并计算金额。
　　（2）订单支付：用户对待支付订单进行支付操作（本系统采用模拟支付），支付成功后才正式扣减库存。
　　（3）订单状态流转：系统需支持待支付、已支付、配送中、已完成、已取消、已退款六种订单状态。
　　（4）退款处理：已支付未发货订单可申请退款，退款成功后恢复库存；已完成订单在实际业务中通常进入售后流程，本文仅作简化处理。
　　（5）超时自动取消：超过30分钟未支付的订单应自动转为已取消状态，不涉及库存回补。
　　（6）角色分视图：普通用户只能看到自己的订单，团长可看到所属活动的订单，管理员可查看所有订单。
　　（7）数据导出：支持将订单列表导出为Excel（CSV）文件。
　　3.1.6 团长管理模块
　　团长管理模块负责团长的申请审核、佣金管理、排行展示和启用/禁用控制。
　　3.1.7 智能分析模块
　　智能分析模块是本系统在管理功能之外接入的运营辅助模块，主要用于为业务决策提供一定的数据支持。具体功能需求包括：
　　（1）销量预测：基于商品的历史销量数据，预测未来3/7/14天的逐日销量，并给出预测可靠性评分。
　　（2）智能推荐：根据用户的购买历史，推荐可能感兴趣的商品，展示推荐分数和推荐理由。
　　（3）用户聚类：将全部用户按消费行为自动分为若干群体，生成群体画像和运营建议。
　　（4）库存补货建议：基于销量预测结果，计算安全库存并按紧急度排序给出补货建议。
　　（5）数据可视化：预测结果以折线图、柱状图、散点图、饼图等形式直观展示。
　　3.1.8 其他功能模块
　　（1）管理后台仪表盘：展示核心KPI、今日数据、销售趋势、订单状态分布和热销排行等运营概览。
　　（2）收货地址管理：支持地址的增删改查，每个用户可设置一个默认地址。
　　（3）购物车：支持商品的添加、删除、数量修改和清空操作，数据持久化到浏览器本地存储。
　　（4）消息通知：系统向用户发送站内消息（订单通知、系统公告、活动通知等），支持已读/未读状态管理。
　　（5）操作日志：自动记录管理员和用户的写操作，支持按模块和用户筛选查看。
3.2 系统非功能需求分析
　　3.2.1 性能需求
　　（1）页面响应时间：普通页面加载时间应在2秒以内，智能分析页面应在3秒以内。
　　（2）并发处理：系统应能支撑至少50个用户同时在线操作。
　　（3）数据库连接池：配置HikariCP连接池，最大连接数20，确保数据库访问效率。
　　3.2.2 安全需求
　　（1）密码安全：用户密码必须经过BCrypt加密存储，禁止明文存储。
　　（2）认证授权：所有需要登录的接口必须验证JWT令牌，管理后台接口需额外验证管理员角色。
　　（3）防注入：使用MyBatis-Plus的参数化查询防止SQL注入攻击。
　　（4）文件上传安全：限制上传文件类型（仅允许图片格式）和大小（不超过5MB）。
　　3.2.3 可用性需求
　　（1）界面友好：采用Element Plus组件库，遵循一致的设计规范，操作直观易学。
　　（2）错误提示：操作失败时给出清晰的错误提示信息。
　　（3）降级处理：智能算法微服务不可用时，主系统应正常运行，仅智能分析功能不可用。
3.3 本章小结
　　本章从功能需求与非功能需求两个层面对系统建设目标进行了分析，明确了核心业务流程、辅助功能模块以及性能、安全和可用性等方面的要求，为后续总体设计与模块实现提供了依据。

4.1 系统架构设计
　　4.1.1 系统总体架构
　　本系统整体采用"客户端层-前端应用层-安全认证层-后端服务层-数据存储层"的分层结构，并在后端之外以独立 Python 服务提供扩展分析能力。系统主业务由 Java 后端承载，智能分析模块通过 HTTP 接口接入，不直接影响核心交易流程。
　　
图4-1 系统总体架构图
　　在客户端层，系统面向 PC 浏览器、手机浏览器以及 H5 访问场景提供统一入口；在前端应用层，采用 Vue 3、Vite、Element Plus 和 Pinia 组织页面、状态与交互逻辑，并通过 RESTful API 与后端进行数据交换。对于运营数据展示部分，则使用 ECharts 完成图表渲染。
　　后端服务层以 Spring Boot 为核心，遵循 Controller、Service、Mapper 的分层组织方式。Controller 负责接收请求与返回结果，Service 负责业务编排，Mapper 负责数据访问；同时配合 JwtInterceptor、AdminInterceptor 和 CorsFilter 实现令牌校验、角色权限控制与跨域处理。对于定时性业务，系统通过 ScheduledTasks 完成超时订单取消和活动状态更新等任务。
　　考虑到预测、聚类与补货等接口涉及经营数据，系统对扩展分析接口实施差异化权限控制：推荐接口面向登录用户开放，预测、聚类与补货建议接口仅向运营人员或管理员角色开放。
　　扩展分析层采用独立部署的 Python Flask 微服务，内部封装销量预测、个性化推荐、用户聚类和补货建议等模块。该层通过 JSON 形式返回分析结果，由 Java 主系统统一对外提供业务接口，从而保持系统职责边界清晰。
　　4.1.2 前后端分离架构
　　前后端分离模式将页面展示与业务处理解耦，使前端能够专注于页面组织和交互逻辑，后端则专注于接口组织、业务规则和数据访问。对于本系统这类管理后台应用而言，该模式更有利于模块拆分、并行开发和后续维护。
　　4.1.3 Java与Python微服务通信设计
　　将智能分析能力设计为独立微服务，而不是直接嵌入 Java 主系统，是本系统在总体设计中的一项重要取舍。若将相关算法直接写入主业务系统，虽然可以减少跨语言调用开销，但会增加主系统的耦合度，并可能在算法异常时影响核心业务流程的稳定性。
　　综合比较后，本文采用 Python 独立微服务方案。通信方式为同步 HTTP 调用：Java 端通过 SmartController、SmartServiceImpl 与 RestTemplate 组织请求，Python 端提供 /api/forecast、/api/recommend、/api/cluster 和 /api/restock 等接口并返回 JSON 结果。该方式虽然会带来一定通信开销，但能够降低算法异常对主业务系统的影响，也便于后续扩展和独立维护。

第5章 核心业务模块设计与实现
　　本章围绕用户认证与权限控制、商品管理、团购活动管理以及订单状态流转等核心模块展开，重点说明各模块在 Java 后端中的设计依据、关键处理流程及其在系统中的作用。
5.1 用户认证与权限控制模块
　　5.1.1 认证方案选型
　　Web应用的身份认证主要有基于Session的有状态认证和基于Token的无状态认证两种方案，对比如表5-1所示。
表5-1 Session认证与Token认证对比
　　对比维度　　Session认证　　Token（JWT）认证　　状态存储　　服务端存储会话信息　　客户端携带令牌，服务端无状态　　水平扩展　　需要Session共享（Redis等）　　天然支持多实例部署　　跨域支持　　依赖Cookie，跨域复杂　　通过HTTP头传递，跨域友好　　微服务适配　　需要统一Session存储　　各服务独立验证令牌即可　　
　　本系统采用前后端分离架构，且后端需要与Python微服务协作。在此背景下，JWT的无状态特性具有明显优势：前端通过HTTP头携带令牌解决了跨域问题，各服务独立验证令牌签名即可确认用户身份，无需共享Session存储。因此选择JWT作为认证方案。
　　5.1.2 安全体系设计
　　系统的认证流程为：用户提交登录凭证后，后端完成身份校验并生成 JWT 令牌；前端将令牌保存于本地状态中，后续请求通过 Axios 拦截器自动携带 Authorization 请求头；后端的 JWT 拦截器对令牌签名与有效期进行校验，并将解析出的用户身份写入基于 ThreadLocal 的 UserContext，以供后续业务逻辑使用。
　　在并发访问场景下，若直接采用共享静态变量保存用户上下文，容易出现线程间身份信息混淆的问题。基于这一考虑，系统最终采用 ThreadLocal 对用户上下文进行隔离，并在请求结束后主动清理相关数据，以保证线程复用条件下的身份安全。
　　密码安全方面选择BCrypt算法。相比MD5和SHA-256等快速哈希算法，BCrypt通过cost factor参数控制计算开销（单次约100ms），并且每次生成随机盐值，即使相同密码也产生不同密文。
　　权限控制采用RBAC模型，定义三种角色。后端通过AdminInterceptor拦截器校验管理员接口权限，前端通过Pinia Store中的计算属性控制菜单显隐，形成前后端双重校验--即使前端代码被篡改，后端仍能拒绝越权请求。
　　5.1.3 前端状态管理
　　前端采用Pinia Store + localStorage的双层方案解决SPA中的登录状态持久化问题：Pinia提供响应式内存状态，localStorage保证页面刷新后状态不丢失。配合Vue Router全局前置守卫（未登录重定向）和Axios响应拦截器（401自动跳转），构成了完整的客户端认证链路。
5.2 商品与分类管理
　　5.2.1 数据模型设计
　　商品的价格字段采用 BigDecimal 类型而非 Double。由于金额计算场景对数值精度要求较高，若直接使用浮点数类型，可能在累计计算过程中引入精度误差，进而影响订单金额的一致性。因此，系统在数据模型设计中采用 BigDecimal 处理价格字段，以保证金额计算的准确性。
　　每件商品设置原价（originalPrice）和团购价（groupPrice）双价格字段，体现社区团购"团购价低于原价"的核心商业逻辑。
图片存储采用文件系统存储+数据库存路径的方案。主图路径存储在mainImage字段，多张详情图以JSON数组格式存储在images字段（如["url1","url2"]），在不增加关联表的前提下实现了一对多关系的简洁表达。
　　5.2.2 查询优化与上传安全
　　商品列表查询利用MyBatis-Plus的LambdaQueryWrapper实现动态条件拼接，配合物理分页和字段索引（categoryId、status），保证了多条件筛选下的查询性能。
　　文件上传安全从三个层面进行防护：类型校验（仅允许image/jpeg、image/png等MIME类型）、大小限制（5MB）和UUID重命名。其中UUID重命名不仅是为了防止文件名冲突，更重要的是防止恶意构造的文件名（如含路径分隔符../的文件名）引发路径穿越攻击。
　　分类模块采用层级树结构，后端递归构建分类树返回嵌套JSON，前端以树形表格渲染。删除操作设置了二次确认机制，防止误操作。
5.3 团购活动管理
　　5.3.1 多对多关系建模
　　团购活动与商品之间为多对多关系，通过关联表（t_activity_product）建立联系。关联表除外键外还存储 activity_price 和 activity_stock 两个业务字段，使同一商品在不同活动中可以设置差异化的价格和库存。该设计有助于将活动配置与商品基础信息分离，增强活动管理的灵活性。
　　5.3.2 生命周期管理
　　为保证团购活动生命周期管理的规范性，系统将活动状态划分为未开始、进行中和已结束三类，并通过分离式接口控制状态流转。该设计有助于在接口层面对非法状态变更进行约束，从而降低业务流程紊乱的风险。
　　前端活动编辑页面需要协调活动信息、时间范围和商品列表三类数据的编辑。时间选择采用日期范围选择器，用户在一次交互中同时选择开始和结束时间，避免"结束时间早于开始时间"的逻辑错误。商品列表使用"弹窗选择+表格行内编辑"的交互模式，活动价格和库存可直接在表格中修改。
5.4 订单管理与状态流转模块
　　5.4.1 订单状态机
　　订单的生命周期以有限状态机建模，定义6种状态及其合法转换路径：
　　订单在业务流程上经历待支付、已支付、配送中、已完成等主要状态，其中未支付订单可取消，已支付且未发货订单可进入退款流程。围绕这些状态变化，系统建立了相应的状态控制逻辑，以保证订单处理过程的一致性和可追踪性。
　　状态转换遵循单向性（终态不可回退）和前置条件校验两个原则。后端为每种转换设计独立接口，并在接口内部封装相应的校验逻辑和时间戳记录，从而使不同状态变化能够对应清晰的业务语义。
　　5.4.2 数据模型与并发安全
　　订单数据采用主表（t_order）+明细表（t_order_item）的一对多结构。收货地址采用快照存储而非外键引用--将收货人姓名、电话和地址直接存储在订单表中。这样做的原因是用户可能在下单后修改或删除地址，外键引用会导致历史订单的收货信息随之变化或丢失。
　　在订单处理过程中，库存扣减的一致性是系统实现中的关键问题。针对并发访问条件下可能出现的超卖现象，本文将库存更新操作下沉到数据库层，通过带条件约束的原子更新语句完成扣减，以提高库存处理的安全性和稳定性。
　　本文采用"支付时正式扣减库存、下单时仅做库存预检"的策略。订单创建阶段不锁定库存，最终是否支付成功以下一阶段的数据库原子扣减结果为准；超时未支付订单仅修改订单状态，不涉及库存回补。退款场景主要针对已支付未发货订单，系统在退款成功后恢复相应库存；对于已完成订单，实际业务中通常应进入售后处理流程而非直接恢复可售库存，本文在原型系统中对此作了简化处理。
　　5.4.3 多角色可见性与数据导出
　　订单可见性根据角色差异化控制：三个不同的查询接口分别返回当前用户的订单、团长所属活动的订单和全部订单。前端根据Pinia Store中的角色信息自动调用对应接口。
　　订单列表支持导出 CSV 文件。为保证导出结果在常见办公软件中的可读性，系统在文件生成时补充 UTF-8 BOM 标记，以减少中文内容在不同客户端环境下出现编码异常的情况。
5.5 本章小结
　　本章围绕四个核心业务模块完成了设计与实现。通过对认证授权、金额处理、状态流转和库存控制等关键问题的处理，系统在业务正确性与实现可维护性方面具备了较为稳定的基础。
　　

第6章 扩展功能与运营支撑模块实现
　　除核心业务功能外，系统还实现了团长管理、购物车、消息通知、操作日志和收货地址等辅助模块。本章主要说明这些模块在整体业务流程中的作用及其实现要点。
6.1 团长管理与审核流程
　　团长是社区团购模式中连接平台与消费者的重要角色。本系统采用"申请-审核"两阶段流程：普通用户提交申请，管理员在后台完成审核。审核通过时需要同步更新团长表中的审核状态与用户表中的角色字段，因此相关操作在同一事务中完成，以保证角色变更过程的数据一致性。前端采用标签页整合"全部团长"和"待审核"视图，便于管理员集中处理审核与启用、禁用操作。
6.2 购物车与收货地址管理
　　购物车数据存储需要在实现复杂度与使用体验之间进行权衡。考虑到本系统为原型系统，且用户通常在固定浏览器环境下完成操作，本文采用基于 Pinia Store 与 localStorage 的前端持久化方案，以满足当前场景下的购物车功能需求。 系统同时支持每个用户维护多个收货地址并设置默认地址，默认地址切换采用"先重置后设置"的处理方式，以保证任一时刻仅存在一个默认地址。
　　购物车项通过商品ID和活动ID的联合标识去重--同一商品在不同活动中被视为不同项，因为活动价格和库存可能不同。数量约束在[1, 库存上限]范围内，从前端层面防止超库存购买。
6.3 消息通知与操作日志
　　系统的消息分为订单通知、系统公告和活动通知三种类型，具有已读/未读状态管理。所有接口通过 UserContext 限定当前用户，确保数据隔离；未读消息计数通过独立接口提供，前端在导航栏展示未读数量提示。
　　操作日志采用 Spring MVC 拦截器（HandlerInterceptor）实现自动采集，在 afterCompletion 阶段记录写操作请求、操作者信息及所属业务模块。该方案能够在不显著增加业务代码耦合度的前提下，为系统审计和问题追踪提供基础支撑。
　　相较于在各个接口中分别手工编写日志逻辑，拦截器方案具有统一处理和维护成本较低的特点。其局限在于日志描述粒度主要依赖请求路径与方法进行归类，但对于本系统当前的审计需求而言已基本满足使用要求。
6.4 本章小结
　　本章围绕团长管理、购物车、收货地址、消息通知和操作日志等辅助功能模块进行了设计与实现说明。这些模块虽然不直接构成业务主线，但对系统完整性、操作便利性和管理可追溯性具有重要意义。
　　

第7章 智能分析扩展模块实现与评估
　　智能分析模块在本文中属于对主业务系统的扩展补充。本章结合社区团购场景，对销量预测、商品推荐、用户聚类和补货建议等功能的实现思路与实验结果进行说明，重点体现其在运营支撑中的辅助作用，而非将其作为论文的主体贡献。
7.1 智能分析微服务设计与数据准备
　　7.1.1 实验数据集说明
　　本章各项实验均基于系统测试运行期间积累的模拟业务数据展开。数据集整体规模为：用户 87 名（其中 62 名有购买记录、43 名有 5 次以上购买记录），商品 32 种（覆盖生鲜、粮油、零食等 6 个分类），订单约 500 条（含订单明细约 1200 条），时间跨度为 30 天。相关结果主要用于验证扩展分析模块在当前场景下的接入可行性与初步应用效果。
　　需要说明的是，当前数据规模与真实大规模业务环境仍存在明显差距，因此本章实验更适合用于观察方法表现与效果趋势，而不宜将其理解为对算法能力的充分证明。
　　7.1.2 数据获取与预处理
　　算法服务通过 PyMySQL 访问 MySQL 业务数据库，并借助 pandas 完成数据清洗与特征构造。为保证与主业务系统的职责分离，Java 端不直接参与算法计算，仅负责触发调用、接收结果并将其组织为业务可用的数据接口。在销量时间序列预处理中，系统使用 pandas 的 date_range 生成连续日期序列，并将无订单日期补零，以保证滞后特征和滑动统计量的计算保持一致。
　　为提高实验可复现性，本文统一采用固定的训练/验证划分策略（前 23 天训练、后 7 天验证）与随机种子设置（random_state=42），并在同一硬件环境下重复运行关键实验 3 次，记录平均值作为最终结果，以降低随机初始化与数据划分波动对结论的影响。
7.2 销量预测方法与实验
　　7.2.1 问题定义与方案选型
　　销量预测的目标是：给定某商品过去30天的每日销量序列，预测未来N天（N=3/7/14）的逐日销量。
　　单一模型往往只能捕捉数据的某一方面特征：线性回归擅长线性趋势但无法处理非线性波动；随机森林擅长复杂模式但容易过拟合短序列；指数平滑擅长序列惯性但对突变反应迟钝。集成学习通过融合多个互补模型，可以获得更稳定的预测结果[3]。基于这一思路，本系统设计了三模型集成方案。
　　7.2.2 三模型集成方案
　　模型一：线性回归（Linear Regression）。该模型通过最小化残差平方和建立输入特征与销量之间的线性关系，具有计算速度快、参数可解释性较强等特点。其基本形式可表示为：
y = β0 + β1x1 + ··· + βpxp	（式7-1）
　　模型二：随机森林回归（Random Forest Regressor）。该方法基于 Bagging 思想集成 100 棵决策树（最大深度为 5），通过对多棵树的预测结果求均值以降低方差。随机森林能够较好地刻画特征间的非线性交互关系，对异常值也具有一定鲁棒性。将最大深度限制为 5，主要是考虑到 30 天短序列样本较少，过深模型可能导致训练集拟合过强而削弱验证集表现，因此本文采用较为保守的参数设置。
　　模型三：Holt 双指数平滑（Holt's Exponential Smoothing）[5]。该方法通过对水平分量和趋势分量进行指数加权更新来预测未来值，适用于具有一定趋势特征的短期序列。其递推关系如下：
Lt = αXt + (1 ? α)(Lt?1 + Tt?1)	（式7-2）
Tt = β(Lt ? Lt?1) + (1 ? β)Tt?1	（式7-3）
Xt+k = Lt + kTt	（式7-4）
　　其中，Lt 表示水平分量，Tt 表示趋势分量，α 与 β 为平滑系数。结合当前样本规模与序列波动特征，本文取 α=0.3、β=0.1。该方法不依赖额外特征工程，可直接基于原始销量序列进行建模。

　　为线性回归和随机森林构建了7个时序特征：
表7-1 预测模型特征设计
　　特征　　计算方式　　捕捉维度　　day_index　　时间序号（0,1,2,...）　　长期趋势　　lag_1　　前1天销量　　短期自相关　　lag_3　　前3天销量　　短期自相关　　lag_7　　前7天销量　　周期性　　rolling_mean_3　　近3天滑动均值　　短期趋势　　rolling_mean_7　　近7天滑动均值　　中期趋势　　rolling_std_3　　近3天滑动标准差　　波动程度

7.3 个性化推荐方法与实验
　　7.3.1 协同过滤方法选型
　　协同过滤的核心假设是利用群体行为数据进行个性化推荐[13]。User-CF从用户维度出发寻找偏好相似的邻居用户，优势在于能发现潜在兴趣；Item-CF从商品维度出发为已购商品寻找相似商品，优势在于推荐结果相关性强。
　　Sarwar 等（2001）指出 Item-CF 在大规模数据下具有较好表现[2]。考虑到本系统当前用户数和商品数均较少，两种方法的计算成本都处于可接受范围，因此本文采用 6:4 的加权方式融合两类结果，以兼顾兴趣发现与商品相关性。
　　7.3.2 相似度度量与融合策略
　　本系统选择余弦相似度作为核心度量：
　　余弦相似度：sim(a, b) = (a·b) / (||a|| × ||b||)
　　余弦相似度衡量向量方向一致性而非绝对大小--用户A购买了10次、用户B购买了2次，只要偏好分布一致，余弦相似度就会给出较高评分，而欧氏距离会因购买量差异低估相似性。
　　推荐流程：从订单数据构建用户-商品购买量矩阵→分别计算User-CF和Item-CF分数→按6:4权重融合并归一化→取Top-N商品并生成推荐理由。对于冷启动用户（无购买记录），Python端返回空列表，Java端以全局热销商品兜底。
　　7.3.3 推荐实验验证
　　采用基于最近一次订单留出的离线评估：对每个有5次以上购买记录的43名用户，按时间顺序保留最近一次订单中的商品集合作为测试集，将此前交互数据作为训练集，并以Top-5推荐列表与测试商品集合的重合情况计算Precision@5、Recall@5和NDCG@5。由于单个订单可能包含多个商品，因此单用户测试集中可能存在多个相关项。

　　7.4.1 特征设计与标准化

　　7.4.2 K-Means与自动选K
　　K-Means需要预先指定聚类数K。本系统采用轮廓系数（Silhouette Score）[4]自动选K，公式为：s(i) = (b(i) ? a(i)) / max(a(i), b(i))，其中a(i)为簇内平均距离，b(i)为最近他簇平均距离。系统遍历K=2到K=6，选择轮廓系数最高的K值。相比Elbow方法，轮廓系数提供了明确的最优K判定标准，避免了"拐点"位置的主观性。

7.5 补货建议生成逻辑
　　库存补货建议是销量预测的业务延伸，形成"预测-决策"闭环。系统对每个商品调用预测模型获取未来 N 天需求量，并乘以安全系数（1.2）得到安全库存量，再与当前库存比较计算缺口。该安全系数主要作为原型系统中的经验参数，用于为运营人员提供相对保守的补货参考。

第8章 管理后台仪表盘模块的设计与实现
　　管理后台仪表盘是运营人员获取平台概览信息的重要入口，其目标是在首页集中呈现关键业务指标和统计结果。
8.1 数据聚合策略
　　仪表盘需要展示多项 KPI 和时序统计数据。为减少前端频繁请求带来的交互开销，系统采用聚合接口模式：后端提供单一的 /api/admin/dashboard 接口，在服务端完成主要数据查询和组装后一次性返回结果。
8.2 数据可视化设计
　　可视化部分采用ECharts实现，图表类型的选择基于数据特性：折线图展示销售额趋势（适合连续时序走势）；柱状图展示订单数量（适合离散时间点比较）；环形饼图展示订单状态占比（适合部分与整体的比例关系）；横向柱状图展示热销Top5（商品名称有足够展示空间）。
　　前端在 onMounted 中于页面完成渲染后初始化图表，并注册 resize 监听器以支持响应式缩放；在 onBeforeUnmount 中移除监听器并释放 ECharts 实例，从而减少页面切换过程中不必要的资源占用。
8.3 本章小结
　　本章围绕管理后台仪表盘的设计与实现进行了说明。通过对关键业务数据的聚合展示与可视化表达，系统能够为管理人员提供较为直观的运营信息支持。

　　使用 Apache JMeter 模拟 10/20/50 个并发用户，每个用户执行"登录→浏览商品→查看订单→访问仪表盘"的典型流程，循环 10 次。该测试主要用于观察常规管理访问场景下系统的稳定性表现。

9.5 系统运行问题与优化
　　功能测试过程中共发现并修复了 3 个典型缺陷：
　　上述缺陷分别涉及前端状态同步、后端事务边界与接口参数校验三个层面。修复后通过回归测试确认相关问题未再次出现，体现了"缺陷定位-修复实施-回归验证"的基本质量保障过程。
　　缺陷1：并发下单时库存超卖（严重）
　　在并发下单场景中，若库存处理采用"先查询、再判断、后扣减"的方式，容易因竞态条件导致超卖。针对该问题，系统将库存扣减修改为带库存约束的原子更新操作，并通过更新影响行数判断扣减是否成功。修复后，在当前测试条件下未再观察到库存为负的情况。
　　缺陷2：JWT过期后前端未跳转登录页（一般）
　　问题描述：令牌过期后继续操作，接口返回 401 但页面未及时跳转。原因是 Axios 响应拦截器缺少对 401 状态码的统一处理。修复方案：在拦截器中增加 401 判断，清除本地状态并跳转登录页。将 JWT 有效期临时调整为 1 分钟进行验证后，问题得到修复。
　　缺陷3：销量预测时间序列缺失日期导致错位（一般）
　　问题描述：部分商品的预测结果出现明显错位。原因是 SQL 的 GROUP BY date 查询结果中不含无订单日期，导致时间序列不连续、滞后特征计算错位。修复方案：通过 pandas 的 date_range 生成完整日期序列，左连接后将缺失日期填充为 0。
　　综上，本轮测试共发现缺陷 3 个（严重 1 个、一般 2 个），均已修复并通过回归验证。
9.6 测试结论
　　综合来看，在当前测试场景与原型环境下，30 个功能测试用例均通过，性能测试结果基本满足预期目标，并发测试表明系统在当前访问规模下能够保持较稳定运行。测试过程中发现的 3 个典型缺陷均已完成修复，其中库存超卖问题的处理对提升系统业务可靠性具有较重要意义。
　　

第10章 总结与展望
10.1 工作总结
　　本文围绕社区团购管理系统的设计与实现开展研究，在梳理业务需求的基础上，完成了系统架构设计、核心功能实现、智能分析模块接入以及系统测试等工作，形成了一套面向社区团购场景的管理系统原型。
　　在业务实现方面，系统覆盖用户、商品、分类、团购活动、订单、团长、地址、购物车、消息通知和操作日志等主要模块，并结合JWT认证、BCrypt密码加密与RBAC权限控制完成了基本的安全设计，能够满足社区团购管理场景下的常见业务需求。
　　在智能分析方面，本文实现了销量预测、协同过滤推荐、用户聚类和补货建议等功能，并在现有数据条件下进行了初步评估。结果表明，相关方法能够在当前系统中提供一定的运营辅助参考，但算法部分整体定位仍以辅助支撑为主。
　　在系统测试方面，本文完成了功能测试、性能测试与并发测试。结果显示，系统在以业务接口为主的访问条件下能够较稳定地运行，但对于包含智能分析接口的混合负载场景，仍有进一步优化空间。
10.2 主要贡献
　　本文的主要工作和贡献可概括为以下三个方面。
　　（1）围绕社区团购场景完成了管理系统原型的设计与实现，形成了较为完整的业务功能框架。
　　（2）在 Java 主业务系统之外接入销量预测、商品推荐和用户聚类等智能分析功能，并完成了与业务系统的协同集成。
　　（3）在当前样本规模和实验条件下，对预测、推荐、聚类及补货建议功能进行了初步验证，为后续围绕运营支撑功能的优化提供了实现基础。
10.3 不足与展望
　　在完成系统设计与实验分析的过程中，本文也存在一些不足，后续仍有进一步完善的空间。
　　（1）数据规模方面。当前实验数据主要来自小样本模拟业务数据，尚不足以全面反映系统在大规模真实场景中的运行效果。后续可接入更长时间跨度的真实运营数据，并结合线上实验进一步评估扩展分析模块的实际收益。
　　（2）推荐方法方面。当前推荐主要基于协同过滤，仍存在冷启动和特征利用不足的问题。后续可考虑引入融合商品内容信息的混合推荐方法，以提高推荐覆盖率和适应性。
　　（3）计算方式方面。当前预测与聚类仍以批处理为主，实时响应能力有限。后续可结合缓存机制或流式处理方式，进一步提升系统的实时分析能力。
　　（4）工程完整性方面。当前系统尚未接入真实支付与物流链路，且移动端场景尚未覆盖。后续可围绕完整交易闭环和多端协同继续完善系统功能。
　　总体而言，本文的研究工作仍处于原型验证阶段。未来可在真实业务环境中进一步检验系统在转化率、复购率和库存周转等方面的实际作用。
